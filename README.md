# ch_hierarchy_study 
# Построение на базе ClickHouse хранилища данных кубов Cognos TM1 для анализа в виде сводной таблицы с иерархическими измерениями

## Контекст

Из кубов Cognos TM1 выгружаются данные, которые имеют вид Измерение1, ..., Измерение N, Value. Каждое Измерение представляет собой набор узловых ("консолидированных") и листовых элементов, которые связаны в множество иерархий, основную (как правило, GTotal) и альтернативные. Все эти иерархии задаются через таблицу связей элемент - родитель. За счет наличия альтернативных иерархий таблица связей элемент - родитель не уникальна для элемента (т.е. элемент может входит в несколько узлов). Число уровней в разных иерархиях может быть разным, разные ветки могут иметь разуню глубину.

## Задача

Построить на Clickhouse витрину, чтобы обеспечить быстрый анализ в виде сводной таблицы с иерархическими измерениями. В сводной таблице нужны следующие функции, привычные пользователям TM1:
- возможность фильтровать как по узлам, так и по листовым элементам
- возможность в сводной таблице детализировать узлы по входящим в них элементам, конкретная альтернативная иерархия задается фильтром
- значения для узлов определяются суммированием значений их потомков

## Проблематика

Материализация всех вариантов узлов и альтернативных иерархий порождает неприемлемое количество записей.

## Задачи MVP

- Загрузка архивированных csv в Clickhouse
- dbt - для преобразований с целью формирования витрины и справочников
- superset - визуализация
- airflow - оркестрация

## Варианты реализации иерархического анализа в Clickhouse

Возникли и рассмотрены нижеприведенные варианты реализации иерархических измерений в Clickhouse и запросов к ним из Superset.

Вариант материализовать сами иерархии в витрину не подходит из-за следующего:
- количество записей в витрине увеличтится до неприемлемого размера
- развертывание материализованного значения узла по составляющим его элементам становится сложной задачей

1. Иерархический словарь (подвариант - отдельный словарь для каждой иерархии)
	- сложности:
		- ключ может быть только одним числовым полем, строго уникальным; поскольку в источнике все ключи измерений строковые, то понадобится суррогатный ключ через хэш и словари для получения строкового ключе на основе суррогатного
		- предполагает только одну иерархию, без альтернативных (а в исходной задаче альтернативные иерархии существуют для почти всех измерений)
	- преимущества:
		- минимальный размер хранения
		- можно использовать встроенные функции Clickhouse для работы с иерархиями
        - словари дают высокую скорость
	- подвариант:
		- для обеспечения уникальности ключей узла/элемента в иерархии заводим отдельный словарь для каждой иерархии
        - самый понятный вариант для реализации и работы в рамках стандарта Clikhouse
        - вариант подходит только тогда, когда альтернативных иерархий не очень много и новые альтернативные иерархии появляются редко
        - неоднозначные иерархии (у которых один и тот же узел/элемент может встречасться несколько раз) не могут быть реализованы таким образом

2. Иерархический словарь (подвариант - суррогатный ключ на базе комбинации Иерахия + Узел)
    - для обеспечения уникальности ключей узла/элемента в словаре берем в качестве ключа узла/элемента хэш от комбинации "Иерахия" + "Узел/Элемент", т.е. как бы делаем альтернативные иерархии отдельными ветками одной большой иерархии с разными пространствами ключей
    - подходит для случая меняющихся альтернативных иерархий
    - неоднозначные иерархии тоже не могут быть реализованы таким образом
    - словарь получается больше, медленнее, а работа с ним сложнее
    - словарь с синтетическим ключом xxHash32({hier:String}||' '|| {el:String})

3. Словарь с развернутой в столбцы иерархией
	- сложности:
		- усложняется фильтрация по узлу/элементу, когда не известно, в каком столбце развернутой иерархии он находится
		- развернутые иерархии могут сформировать объемный словарь
        - неоднозначные иерархии тоже не могут быть реализованы таким образом (поскольку dictGet для листового элемента требует однозначности)
	- преимущества:
		- простота аналитических запросов
		- скорость доступа
    - используется составной ключ "Иерахия" + "Узел/Элемент"
	- подварианты реализации:
		- для каждого уровня своя колонка
        - массив вышестоящих элементов вместо набора полей
    - здесь и далее буду рассматривать только массивы, поскольку:
        - структура хранения не требует переделки при появлении иерархии с большой глубиной
        - уровень i удобно получать как arr[i], при этом для несуществующих уровней Clickhouse автоматически вернет пустое значение
        - удобно получать "общий путь" как ```arrayStringConcat(arr[, separator])```
4. MergeTree таблица с развернутой в столбцы иерархией
	- вариант аналогичен варианту словаря с развернутой в столбцы иерархией с той лишь разницей, что вместо словаря используется таблица MergeTree
	- по сравнению со словарем:
		- иерархия может быть более объемной, а работа с ней более универсальной
        - для запросов нужен join, что может быть медленно
		- иерархии могут быть неоднозначными, т.е. включать один и тот же листовой элемент в разных ветках; при этом для неоднозначной иерархии (INNER JOIN) оптимизатор не сможет не выполнять JOIN, поскольку это влияет на сумму

5. Таблица с движком Join для развернутой в столбцы иерархии
	- вариант аналогичен варианту словаря с развернутой в столбцы иерархией (и MergeTree таблицей) с той лишь разницей, что вместо словаря/MergeTree используется таблица с движком Join
	- по сравнению со словарем:
		- иерархии могут быть неоднозначными, т.е. включать один и тот же листовой элемент в разных ветках; при этом для неоднозначной иерархии (INNER JOIN) оптимизатор не сможет не выполнять JOIN, поскольку это влияет на сумму
        - работать с такой таблицей сложнее, чем со словарем
	- по сравнению с MergeTree
		- по идее быстрее, поскольку уже таблица уже в памяти в готовом для join виде
		- всегда в памяти, что накладывает ограничения на объем таблицы
        - таблица может использоваться только для Join по заданным при создании ключам, другие запросы к ней не возможны
        - использование такой таблицы сложно и ограничено

### Выводы

Как ни странно, лучшую скорость показал вариант с MergeTree таблицей развернутой иерархии. Он же и самый универсальный. Видимо, для таких таблиц оптимизатор выдает лучшую производительность. Однако при этом потребляется на 30% больше RAM и на 20% больше чтения диска.

Чуть более медленной, но менее требовательной к ресурсам альтернативой является использование иерархических словарей CLickhouse.

Вариант же использования движка Join не имеет смысла, с ним сложно работать и оптимизатор не сокращает таблицу Join перед самим джойном, что приводит к плохой производительности.

## Реализация экстракции, загрузки, трансформации данных в Clickhouse

Для загрузки из S3 используется движок S3. Создается через специальную материализацию dbt - external_table. См. dbt/macros

Витрина через dbt материализуется как incremental, инкремент отслеживается по полю _time - время загрузки в S3.

Измерения через dbt материализуются как таблицы (table) или словари (dictionary).

Для формирования нужной структуры таблиц и словарей используется WITH RECURSIVE. См. dbt/models

Airflow испольузет dag на основе dbt с помошью пакета cosmos. См. dags

## Реализация анализа в Superset

В Superset для обеспечения выбора узла иерархии используется датасет d_g_production_groups_h. См. superset/datasets.

В основных датасетах используется Jinja, чтобы:
- обеспечить применение выбранной иерархии (выбранной на датасете d_g_production_groups_h)
- отфильтровать по выбранному узлу, при выбора иерархии целиком фильтр по узлу не добавляется
- если никакая иерархия  не выбрана, то не выполнять ни джойнов, ни функций словарей, заполнять знапчение листовым элементом из витрины.

См. superset/datasets
